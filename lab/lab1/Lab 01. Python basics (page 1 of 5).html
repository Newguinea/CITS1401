Information sections and exercises

You will often find the same information from the lectures here, but learning to program is about trying things for yourself (just like riding a bicycle). The information sections will usually include suggested Python exercises which will help you to develop skills you need to answer the questions correctly. Don't just skip these information exercises. If you don't do them you will miss the opportunity to learn, and you will risk losing marks when you attempt the quiz questions (and further, your other assessments and exams).

The information sections and suggested Python exercises are carefully constructed to point out key things you need to know about Python. They help you to build up your understanding of some of the more complicated aspects of the tasks the questions ask you to do. The questions are then testing that understanding. You are missing the point of the lab if you try to go straight to the questions. The marking regime will penalise incorrect attempts (as explained in Lab 00) so you will gain better marks by working through the exercises and being able to answer questions right the first time.

Lab questions

If the question requires you to submit code, develop and test your code first in Thonny IDE (you can download is from www.thonny.org) and then copy and paste it into the question answer box. You will incur needless penalties for silly mistakes if you try to type the code directly into the answer box and submit it without trying it in Thonny.

You will then check that the code gives the right answers when executed with testing cases. 

Provided you've checked each question as you've done it, you can leave the lab (logging off the machine) at any time and then later resume the lab until the lab close date/time. However, your answer is only saved when you click the question's Check button. Unless you have clicked Check you will lose your answer if your Moodle Server session automatically times out after a period of inactivity. Follow these simple rules for each coding question:

  • Always develop and test your code in Thonny.
  • In Thonny, save your Python code to one of your own folders.
  • Carefully copy your code to the lab answer box.
  • Click Check to have your submission to that question tested and graded. Remember there will be some hidden tests to test your codes too.
  • Repeat the above steps until your code passes all the tests.

Submitting your whole attempt for marking

Remember to click Submit all and finish button at the end of the lab (not each question as you have one chance of submission only). All open submission will be automatically submitted at the deadline.

Evaluating expressions in the Python Shell

This section introduces you to the Python Shell. Experiment it yourself to improve your learning.

The Python Shell is a "hot line” to the Python Interpreter, i.e., to the program that executes Python programs. The ">>>” that it displays after the introductory lines is a "prompt” for input from the keyboard. 

Lines that you type into the Python Shell get sent to the Python interpreter when you hit Enter and the response from the interpreter is immediately displayed in the Python Shell window followed by a new prompt. This is great for testing short pieces of code, but once you close Thonny, all data in the shell is lost. So if you want to create a larger program or save your Python code, it would be better to create a new Python file and save that. The Python Shell capability will be used throughout labs to introduce some of the fundamental concepts of Python. 

Type each of the following input lines into the shell window, study the response, read the note in the table and try to understand what happened before proceeding. If you can't figure it out, try typing other similar inputs, read the textbook or ask lab facilitator. Answer the quiz questions as you go. Note that in the Python Shell you can use the up-arrow key, multiple times if necessary, to re-enter previous lines.

Although many of the exercises have been demonstrated in the lectures and you may think you know exactly what will happen, you are strongly advised to carry out each one anyway, even if it only proves you right. There is no substitute for first-hand experience.

Input Line

Notes

2 + 3 * 5

If you enter an arithmetic expression, Python evaluates the expression and prints the answer. The '*' is the multiplication operator. It has a higher precedence than '+', so is done first here.

(2 + 3) * 5
Parentheses ("brackets”) are used to change the order of evaluation.
2+3       *       5
Space characters are generally ignored in expressions. However, for readability you should generally type one space character either side of each operator ('+', '-', '*', etc) as in the first two examples, as stated by the Python style guide.
17 / 10
'/' is the divide operator. [If you got the answer 1, then you must be using an earlier version of Python. If you are working at home, please upgrade to Python 3, which is used throughout this course.]
17 // 10
'//' is the integer division operator. When applied to two integers ("whole numbers”), like 17 and 10, it has the kindergarten meaning of "how many times does 10 go into 17”. The answer is "1" with 7 left over.
17 % 10
'%' is the remainder operator - it lets you find the "bit left over”.
10 ** 5
'**' is the exponentiation operator. Read this as "10 to the power of 5”.
10 ** 20 + 1
Python can cope well with large numbers.
10 ** 10 ** 5 - 1
Integers can get very large (and slow to compute!) but are always exact.
2.0 + 3.0 * 5.0
Numbers with decimal points in them are floating point numbers ("floats"). They are approximations of real numbers.
2.0 / 3.0
Floats have only about 16 digits of accuracy. After that you get junk.
10.0 ** 20 + 1.0
A float can't exactly represent the answer to an expression like this. The answer should be 100,000,000,000,000,000,001 but can't be expressed with only 16 digits.
